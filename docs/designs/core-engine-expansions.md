## Vision

The approved plan delivers a typed message bus and module base class. The expanded
version turns TycheEngine into a **complete solo quant operating system** — where the
same strategy code runs identically in backtest (SimClock at 1000x speed) and live,
latency visibility is built-in without external tooling, new strategies are generated
from a CLI in 10 seconds, and the health of every running process is visible from a
browser without touching logs. The engineering foundation is identical to the current
plan — the expansions are additive, not architectural.

## Feature Flag Pattern Decision

Two activation patterns are used across expansion tasks. This is the canonical rule:

- **Config flags** (`metrics_enabled = true` in TOML) — for features that affect
  runtime behavior and must survive process restart (e.g. latency accumulation).
- **Env vars** (`TYCHE_SCHEMA_CHECK=1`) — for dev-only checks that are
  performance-sensitive and should never be set in production (e.g. type enforcement).

All future tasks follow this pattern. No mixing.

### Task 16: RecordingModule + ReplayBus

**Files:** `tyche/backtest/recording.py` (contains both `RecordingModule` and `ReplayBus`).
The `tyche/backtest/` package is created by this task (add `tyche/backtest/__init__.py`).

**Prerequisite (Task 16, Step 0):** Modify `Module.__init__` to accept an optional
keyword-only `clock` parameter: `def __init__(self, nexus_address: str, bus_xsub: str,
bus_xpub: str, *, clock=None)`. Set `self._clock = clock if clock is not None else LiveClock()`
inside `__init__`. `LiveClock` is imported in `tyche/core/module.py` at module level —
`clock.py` must not import from `module.py` (confirm acyclic).
`clock` is a **constructor-only parameter** — it is not added to `ModuleConfig` and not
stored in TOML. Generated config templates are unaffected.

**Clock scope:** Only data/dispatch timestamps use `self._clock.now_ns()`. Heartbeat
scheduling in `Module.run()` (the `next_hb` interval check) must **remain on `time.time()`**
(wall clock). Replacing heartbeat timing with `self._clock` would cause `SimClock`-driven
modules to either never send heartbeats (if `SimClock` is not advanced) or spam them (if
advanced in large increments), silently breaking Nexus liveness tracking in backtest.
Only these calls are replaced with `self._clock.now_ns() / 1e9`: message `timestamp_ns`
fields set inside `_dispatch()` or data handlers.
This is a modification to a plan-approved file (Task 14 output) and **must be flagged
as a spec deviation in the impl log**.

**RecordingModule** — constructor: `RecordingModule(file_path: str, nexus_address: str,
bus_xsub: str, bus_xpub: str)`. Opens `file_path` for **binary append** on `on_start()`
(`open(file_path, "ab")`). Existing file content is preserved; no truncation. The parent
directory must pre-exist (RecordingModule does not call `os.makedirs`). In `on_start()`,
after socket setup, subscribes with empty-string prefix by calling
`self._sub_sock.setsockopt(zmq.SUBSCRIBE, b"")` directly —
**bypassing `TopicValidator.validate()`** (empty string is not a valid topic but is a legal
ZMQ subscription). Do not call `self.subscribe("")`.

**Override `_dispatch()`, not `on_raw()`.** The base `Module._dispatch()` routes known
dtypes (QUOTE, TICK, TRADE, BAR) to typed handlers and only falls back to `on_raw()` for
unknown dtypes — meaning `on_raw()` would silently miss all typed market data messages.
To record ALL messages regardless of dtype, RecordingModule overrides
`_dispatch(self, topic: str, raw_frames: list)`, writes the MessagePack record to file
(reading `wall_ns = time.time_ns()` at the top of `_dispatch()` before the super call),
then calls `super()._dispatch(topic, raw_frames)`. Do not override typed handlers.

**`.tyche` file format:** Each record is a 4-element MessagePack array:
`[topic: str, timestamp_ns: int, payload: bytes, wall_ns: int]`, written
sequentially (MessagePack is self-delimiting; reader uses `msgpack.Unpacker(raw=False)`
so topic strings deserialize as `str`, not `bytes`).
`wall_ns` is **always `time.time_ns()`** (unconditional wall clock), never `self._clock.now_ns()`.
This ensures replay files can be correlated with external timestamps regardless of clock injection.

**ReplayBus** — `ReplayBus` is a **plain class, not a Module subclass**. It does not
register with Nexus. Constructor: `ReplayBus(file_path: str, bus_xsub: str,
speed: float = 1.0)`. `speed=0.0` means publish as fast as possible (no inter-message
delay). `speed=1.0` means wall-clock timing. `speed=10.0` means 10x accelerated.

`ReplayBus.__init__` creates `self._ctx = zmq.Context()`. `ReplayBus.run()` is a
**blocking method** that creates a `zmq.PUB` socket on `self._ctx`, connects to `bus_xsub`,
sleeps briefly (e.g. `time.sleep(0.05)`) for subscription propagation, then reads all records
in order and publishes each. Each message is published as a 3-frame multipart message matching the Bus
envelope spec (design spec §5.4): `[topic_bytes, timestamp_ns_big_endian_8bytes, payload_bytes]`.
`topic_bytes` is `record[0].encode()`. `timestamp_ns_big_endian_8bytes` is `record[1].to_bytes(8, "big")`.
`payload_bytes` is `record[2]` (raw bytes from the replay record, unchanged).
The method sleeps between records per the delay formula and returns when the file is
exhausted. Before returning, `run()` calls `sock.close()` then `self._ctx.term()`.
It does not loop; a single call replays the file exactly once.

**Inter-message delay formula:**
`delay_s = (record[3] - prev_wall_ns) / speed / 1_000_000_000`
where `record[3]` is `wall_ns` from the `.tyche` record and `prev_wall_ns` is the
`wall_ns` of the preceding record. For the first record, delay is 0. Guard:
```python
if speed == 0.0:
    pass  # no sleep — publish as fast as possible
else:
    delay_s = (record[3] - prev_wall_ns) / speed / 1_000_000_000
    if delay_s > 0:
        time.sleep(delay_s)
```

**Tests:** `tests/integration/test_backtest_replay.py`. Each Bus is started as a
`threading.Thread(target=bus.run, daemon=True)` using the `Bus` class, matching the fixture
pattern in `test_module_e2e.py`. First Bus on test ports `35556`/`35557` (distinct from
production and existing test fixtures); second Bus on ports `35558`/`35559`. Test publishes
N messages to the first Bus, RecordingModule writes to a `.tyche` temp file. ReplayBus
reads the file and re-publishes to the second Bus. A subscriber module asserts it receives
the same N messages in the same order, with payload bytes (Frame 2) byte-equal.

### Task 17: Dispatch latency instrumentation

**Prerequisite sequencing:** Task 16 Step 0 (adding `clock` parameter to `Module.__init__`)
must be completed and all Task 16 tests must pass before Task 17 begins. Both tasks modify
`Module.__init__` / `ModuleConfig`; applying them in sequence prevents merge conflicts.

**LatencyStats** — a compact struct in `tyche/utils/latency.py`. Implementation:
a fixed-size ring buffer of 1024 `int64` samples — stored as a `bytearray` of
`1024 * 8 = 8192 bytes` (8 bytes per sample, little-endian signed 64-bit integer).
Each sample is a signed 64-bit integer written using `struct.pack_into('<q', buf, (_count % 1024) * 8, ns)`
(avoids temporary allocation from slice assignment). A `_count` field tracks total samples
written (unbounded). The active buffer length for percentile calculations is
`active_len = min(_count, 1024)`. The ring write index is `_count % 1024`.

Percentile method signature: `def percentile(self, p: float) -> int` where `p` is a
fraction in `[0.0, 1.0)`. Calling with `p >= 1.0` raises `ValueError("p must be in [0.0, 1.0)")`.
The caller passes values such as `0.50`, `0.95`, `0.99`. Calculation: unpack the active
slice as `values = list(struct.unpack_from(f'<{active_len}q', buf))` (reads from offset 0,
order-independent for percentile purposes — percentile is computed over all active samples
regardless of ring position), sort `values`, return `values[min(int(p * active_len), active_len - 1)]`.
Memory: 8 KB fixed. Never grows.

**Module integration:** `Module` stores latency stats as `self._latency: dict[str, LatencyStats]`
initialized to `{}`. A new `LatencyStats()` is created lazily on first sample for each `dtype`
key. The STATUS handler reads `{key: {"p50": s.percentile(0.50), "p95": s.percentile(0.95),
"p99": s.percentile(0.99)} for key, s in self._latency.items()}` when `metrics_enabled` is true.

**Keying:** Keyed by `dtype` string constructed from the topic as follows:
- For 4-segment topics (`CRYPTO_SPOT.BINANCE.BTCUSDT.QUOTE`): key = `parts[3]` (e.g. `"QUOTE"`)
- For 5-segment BAR topics (`CRYPTO_SPOT.BINANCE.BTCUSDT.BAR.M5`): key = `f"BAR.{parts[4]}"` (e.g. `"BAR.M5"`)
- All other topics: key = `parts[-1]`

Bounded key space, directly actionable for performance diagnosis. Not keyed by full topic.

**Activation:** Config flag `metrics_enabled = false` (default off) in `ModuleConfig`.
`ModuleConfig` gains the field `metrics_enabled: bool = False`. `ModuleConfig.from_file()`
must be updated to pass `metrics_enabled=m.get("metrics_enabled", False)` in its
constructor call. Because the field has a Python default, the approved plan's
`example_strategy.toml` (Task 1, Step 6) does NOT need to be updated — the Task 10 config
test (`test_module_config_loads`) continues passing without modification.
The generated `config/modules/<name>.toml` template (Task 18) includes `metrics_enabled = false`
in the `[module]` section.

**Timer placement in `_dispatch()`:** Start timer at the top of `_dispatch()` (before
the `dtype` key extraction and handler lookup); stop immediately after calling the
handler method (not wrapping `on_raw()`). Record elapsed ns into `self._latency[dtype]`.
When `metrics_enabled` is false, skip the timer entirely (no `time.time_ns()` call, no
dict lookup, no allocation).

This is a modification to plan-approved Task 14 output (`ModuleConfig` and `Module._dispatch()`)
and **must be flagged as a spec deviation in the impl log**.

**STATUS response extension:** When `metrics_enabled = true`, `STATUS` command response
includes `dispatch_latency_ns: { "QUOTE": { p50, p95, p99 }, ... }`.

**Tests:** `tests/unit/test_latency_stats.py` — verify ring buffer overflow behavior,
percentile correctness at N=1, N=1024, N=2048 (overflow case).

### Task 18: Strategy scaffold CLI

**Output layout:**
```
strategies/
  <name>.py          ← generated Module subclass
config/modules/
  <name>.toml        ← generated config file
```

The `strategies/` directory is created at the project root if absent. Strategies are
not part of the `tyche/` package — they are user code.

**CLI:** `tyche/cli/__main__.py` implementing `python -m tyche new-strategy <name>`.
Uses stdlib only. Template strings, no external templating library. The CLI writes
`strategies/<name>.py` and `config/modules/<name>.toml` relative to `os.getcwd()`.
Both directories are created if absent (`os.makedirs(..., exist_ok=True)`).

**Strategy template** (`strategies/<name>.py`):
```python
from tyche.module import Module
from tyche.model import ModuleConfig

class {ClassName}(Module):
    """Auto-generated strategy scaffold."""

    def on_start(self):
        pass  # subscribe to topics here

    def on_stop(self):
        pass

    def on_quote(self, topic, msg):
        pass

    def on_bar(self, topic, msg):
        pass

if __name__ == "__main__":
    cfg = ModuleConfig.from_file("config/modules/{name}.toml")
    {ClassName}(cfg.nexus_address, cfg.bus_xsub, cfg.bus_xpub).run()
```
`{ClassName}` is `name` converted to PascalCase (split on `_`, capitalize each word, join).

**Config template** (`config/modules/<name>.toml`):
```toml
[module]
service_name = "{name}"
nexus_address = "tcp://localhost:5555"
bus_xsub = "tcp://localhost:5556"
bus_xpub = "tcp://localhost:5557"
cpu_core = null
metrics_enabled = false
```
Note: `service_name` matches the key read by `ModuleConfig.from_file()` in Task 10.
The strategy template's `{ClassName}(cfg).run()` call is valid because `Module.__init__`
takes `nexus_address`, `bus_xsub`, `bus_xpub` directly — but the scaffold should pass
config fields explicitly: `{ClassName}(cfg.nexus_address, cfg.bus_xsub, cfg.bus_xpub).run()`
so the `ModuleConfig` is not passed as a positional argument. The `clock` parameter is
keyword-only and defaults to `LiveClock()` — not passed in the scaffold.

**Tests:** `tests/unit/test_cli_scaffold.py`. Verify: (a) generated `.py` file passes
`py_compile.compile()`, (b) generated file is importable via `importlib.import_module()`
with `tyche_core` pre-patched via `sys.modules` mock (no Nexus/Bus required at import
time), (c) generated `.toml` loads via `tomllib` without error.

### Task 19: Nexus health dashboard

**Thread model:** HTTP server runs in a `daemon=True` thread calling
`HTTPServer.serve_forever()`, started during `Nexus.run()`. The ZMQ poller thread
and HTTP thread share the registry dict.

**Thread safety prerequisite:** Before adding the HTTP handler, migrate the Nexus
registry from a plain `dict` to a `dict` protected by `threading.RLock`. The poller
thread acquires the lock on every registry mutation (REGISTER, DEREGISTER,
heartbeat update); the HTTP handler acquires the lock on each `GET /status`.

**Endpoint:** `GET /status` on configurable port (default `9999`). Task 19 **replaces**
the Task 1 `config/modules/nexus.toml` with an updated version that adds a `[module]`
section (if absent) and the new `[nexus]` section. Complete updated
`config/modules/nexus.toml` (all keys that must be present in the final file):
```toml
[module]
name = "nexus"
cpu_core = 0

[policy]
heartbeat_interval_ms = 1000
missed_heartbeat_limit = 3
registration_timeout_ms = 5000
registration_max_retries = 3

[nexus]
http_port = 9999
```
All `[policy]` keys from the Task 1 approved file are preserved; only the new `[nexus]`
section is additive. `NexusPolicy.from_file()` is unaffected for its existing fields.

`NexusPolicy` dataclass gains one field: `http_port: int = 9999`. `NexusPolicy.from_file()`
must be updated to pass `http_port=data.get("nexus", {}).get("http_port", 9999)` in its
constructor call (the `[nexus]` section is a top-level TOML section, not nested under
`[policy]`). No new config class is needed. A test is added to `tests/unit/test_config.py`
verifying that `http_port` loads from the `[nexus]` section and that the default (`9999`)
is used when the section is absent.

**Status thresholds** (derived from design spec §4.1 — 3 missed heartbeats = dead):
- `ALIVE` — `missed_hbs == 0`
- `STALE` — `1 <= missed_hbs <= 2`
- `LOST` — `missed_hbs >= 3`

Response schema:
```json
{ "modules": [ { "name": str, "cpu_core": int|null, "status": "ALIVE"|"STALE"|"LOST",
                  "last_hb_ns": int, "missed_hbs": int } ] }
```
Error response: HTTP 503, body `{"error": "registry_unavailable"}` if the lock
cannot be acquired within 100ms. In code: `lock.acquire(timeout=0.1)` (seconds, not ms).
Response includes `Content-Type: application/json` header.

**Shutdown:** `Nexus.stop()` calls `self._http_server.shutdown()` before returning.
`HTTPServer.shutdown()` blocks until `serve_forever()` exits. To avoid up to 500ms
of shutdown latency per test (the default `poll_interval=0.5s`), start the server with
`server.serve_forever(poll_interval=0.05)` — this limits shutdown wait to ~50ms.
Task 19 Step 0 modifies `Nexus.registry` as implemented in Task 13.

**init_ffi_bridge signature:** The approved plan's one-argument form
`init_ffi_bridge(service_name: str)` is the canonical contract for all expansion
tasks. The design spec §6.4 mention of a second `zmq_context_ptr` argument is
aspirational and does not apply until a shared-memory migration.

**Tests:** `tests/unit/test_nexus_http.py` — start Nexus with a registered mock
module, query `/status`, verify JSON schema and module list accuracy.

### Task 20: Topic schema registry

**Activation:** Env var `TYCHE_SCHEMA_CHECK=1` (dev only; never set in production).
When unset, `TopicSchema` is a no-op that returns immediately — zero overhead.
The schema registry feature is required to be implemented; it is not optional. If the
recursive matcher is unexpectedly blocked, log a CRITICAL open item in the impl log
rather than silently skipping.

**Pattern language:** `*` matches exactly one dot-separated segment. `#` matches one
or more trailing dot-separated segments. Examples:
- `*.*.*.QUOTE` matches any 4-segment topic ending in `QUOTE` (e.g. `CRYPTO_SPOT.BINANCE.BTCUSDT.QUOTE`)
- `#` as a standalone pattern matches all topics (any number of segments)
- `*.*.*.BAR.#` matches any BAR topic regardless of interval: matches `CRYPTO_SPOT.BINANCE.BTCUSDT.BAR.M5` and `CRYPTO_SPOT.BINANCE.BTCUSDT.BAR.M15` (the `#` consumes one or more trailing segments)
- `#` only matches at the end of a pattern and consumes **at least one segment** (unlike RabbitMQ convention where `#` can match zero segments — that behavior is NOT used here)

Matching is case-sensitive. Recursive descent pseudocode:
```
match(pattern_parts, topic_parts):
  if pattern_parts is empty: return topic_parts is empty
  if pattern_parts[0] == "#":
    # must consume at least one segment
    return len(topic_parts) >= 1 and (
      match(pattern_parts[1:], topic_parts[1:]) or   # consume one
      match(pattern_parts, topic_parts[1:])           # consume more
    )
  if pattern_parts[0] == "*": return match(pattern_parts[1:], topic_parts[1:])
  return pattern_parts[0] == topic_parts[0] and match(pattern_parts[1:], topic_parts[1:])
```
No regex, no external deps.

**Files modified:** `tyche/utils/topics.py` (add `TopicSchema` class and `_SCHEMA` dict),
`tyche/core/module.py` (add `TopicSchema.check()` call in `publish()`).

**Implementation:** `TopicSchema` class in `tyche/utils/topics.py`, with
`_SCHEMA: dict[str, str] = {}` as a class-level attribute. `_SCHEMA` keys are pattern
strings (not valid topic strings — they must NOT be passed through `TopicValidator.validate()`).
A module-level `_SCHEMA` dict maps pattern strings to expected Python type names (strings, not type
objects, to avoid circular imports). `module.py` imports `TopicSchema` from `tyche.utils.topics`;
this is safe because `topics.py` does not import from `module.py`.
Only `Module.publish()` calls
`TopicSchema.check(topic, type(payload).__name__)` when `TYCHE_SCHEMA_CHECK=1` —
`Module.subscribe()` is **not** a call site (subscribe has no payload to type-check).
`check()` is a `@staticmethod`. `TypeError` raised on mismatch with message:
`"Topic '{topic}' expects {expected}, got {actual}"`. If no pattern matches the
topic, `check()` returns without error (unknown topics are not schema violations).
This is a modification to plan-approved Task 14 output (`topics.py` and `module.py`) and
**must be flagged as a spec deviation in the impl log**.

**Default schema** (registered at import time). Scoped to market data types defined
in the core engine plan — OMS/execution types (`PyOrder`, `PyOrderEvent`, `PyAck`,
`PyPosition`, `PyRisk`) are out of v1 scope and are NOT included; adding them now
would couple the core engine to sub-projects not yet designed:
```python
_DEFAULT_SCHEMA = {
    "*.*.*.TICK":  "PyTick",
    "*.*.*.QUOTE": "PyQuote",
    "*.*.*.TRADE": "PyTrade",
    "*.*.*.BAR.*": "PyBar",
}
# Registration call at module level — runs once on import:
TopicSchema._SCHEMA.update(_DEFAULT_SCHEMA)
```
`payload` at the `Module.publish()` call site is always a `tyche_core` Rust wrapper
type (e.g. `PyQuote`), never a plain Python dataclass alias — `type(payload).__name__`
therefore returns `"PyQuote"` and matches `_DEFAULT_SCHEMA` entries correctly.

**Tests:** `tests/unit/test_topic_schema.py` — verify pattern matching, type
enforcement, TypeError message format, no-op behavior when env var is unset,
and that an unregistered topic does not raise.

2. **Nexus HTTP thread safety:** Registry migrated to `threading.RLock`-protected dict
   in Task 19 prerequisite step. HTTP handler acquires lock per request with 100ms
   timeout; returns 503 if unavailable.

3. **SimClock + ZMQ timing:** Resolved by Task 16 prerequisite step — `Module.__init__`
   accepts `clock` (keyword-only, default `LiveClock()`). `ReplayBus` injects `SimClock`.
   **Heartbeat scheduling stays on `time.time()` (wall clock).** Only data/dispatch
   `timestamp_ns` fields use `self._clock.now_ns()`. Replacing heartbeat timing with
   a simulated clock would break Nexus liveness tracking in backtest. This is a
   modification to plan-approved Task 14 output and must be flagged in the impl log.
